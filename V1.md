# Canon V1 Roadmap

Status: **active**. This document defines what "V1" means for Canon and
tracks the milestones to get there. It supersedes the per-feature slice
plans only in scope — `WASI-HTTP-HANDLER.md` and `STREAMING.md` remain
the detailed designs for their slices and are referenced below.

## What V1 means

Canon V1 is the release where the language's promise — a canonical,
small-surface language that compiles to standard WebAssembly components
and thereby plugs into the whole WASI ecosystem — is actually true, not
aspirational. Concretely, five claims must hold:

1. **Sound surface.** Every program the checker accepts runs correctly.
   No silent codegen gaps: a feature is either rejected at check time
   with a clear message, or it works at runtime.
2. **Real WASI citizenship.** Canon components run on standard hosts
   (`wasmtime serve`, `wasmtime run`, Spin, …), not only inside
   `canon run`. The bindgen covers the practically useful WASI 0.3
   surface instead of skipping most of it.
3. **A complete-enough language.** The core vocabulary a working
   programmer needs exists and runs: collections you can iterate and
   index, booleans you can compose, errors that short-circuit,
   dispatch over any union arity.
4. **A flagship app.** One real application in `examples/` that a
   newcomer can read, run on a stock WASI host, and copy as a starting
   point.
5. **Docs that match reality.** The mdBook, `DESIGN.md`, and
   `CLAUDE.md` describe the implementation as it is; superseded design
   docs are deleted.

## Current state (2026-07-02 audit)

- Test suite: green (`cargo test`, all layers).
- Bindgen emits only ~20 WASI functions total; everything involving
  resources, narrow ints, `option`/`result` shapes, or `list<T>`
  returns is skipped. This is the single biggest gap between the
  pitch and the product.
- All three `wit-component` prototypes have landed and prove that
  routing codegen through `wit_component::ComponentEncoder` removes
  the need for hand-rolled canonical-ABI lowering of resources,
  narrow ints, options, results, and streams.
- `examples/` holds small demos (http-server, fetch-url, read-file,
  random, clock, multifile); no flagship app.
- mdBook docs exist and are structurally sound (getting started, tour,
  reference); need a pass once V1 features land. `DYNAMIC-HANDLERS.md`
  is superseded but still present.
- Language gaps (from the CLAUDE.md gap table): `list.map` is the
  identity, no `list.get`, `Bool.and/or/not` break codegen, 3-variant
  dispatch picks the wrong arm, `?` doesn't short-circuit on `Err`,
  `Int(1)` explicit constructor breaks codegen, nested constructors
  clobber `alloc_ptr` in `build_product_value`, `ToJson` misses
  newtype receivers.

## Milestones

Ordering rationale: M1 is the keystone — it converts a dozen tracked
lowering gaps into non-issues and unblocks M3 and M5. M2 is independent
of M1 and can interleave. M4 runs throughout; M6 is last so it
documents what actually shipped.

### M1 — Component-model backbone (`ComponentEncoder` routing)

The `WASI-HTTP-HANDLER.md` slice 1b + `STREAMING.md` slice 1b work.
Route programs whose entry is HTTP-shaped (and later, any program whose
surface mentions resources/streams) through
`wit_component::ComponentEncoder` instead of the hand-rolled
`wasm-encoder` component sections. The CLI world keeps the existing
path.

- [x] Detect the HTTP entry in codegen via `ast::entry_world_of`.
- [x] Emit core module with `<iface>#<func>` names, embed WIT metadata
      via `embed_component_metadata`, encode with `ComponentEncoder`.
- [x] Hard-coded 200 handler end-to-end
      (`tests/wasi_http_service_test.rs`: compile → `canon run --addr`
      → TCP request → assert 200).
- [x] Remove the "codegen not yet implemented" diagnostic; promote the
      pending checker fixture to `ok/`
      (`tests/checker/ok/wasi_http_handler.can`).

**Exit criterion (met 2026-07-02):** a Canon `(Request) -> Response`
program compiles to a standard `wasi:http/service` component and
serves HTTP under `canon run --addr` (which hosts it through the same
`wasi:http/handler.handle` export any compliant host uses).

### M2 — Language-core correctness (close the silent-wrong gaps)

Each item is a self-contained fix pinned by a new runtime fixture.
Priority order = how silently wrong the current behaviour is:

- [x] `build_product_value` nested-constructor `alloc_ptr` clobber
      (fixed via operand-stack base copies — scales past the
      scratch-local approach; `tests/runtime/product_nested_constructor.can`).
- [x] 3-variant (N-variant) union dispatch (was already fixed; pinned
      by `tests/runtime/union_four_variants_mixed.can`). Found & fixed
      two real `Float` holes instead: `.print()` silently dropped f64s
      and f64 payloads emitted invalid wasm (`float_print.can`,
      `float_in_product.can`).
- [x] `list.get(i)` builtin (`tests/runtime/list_map_get.can`).
- [x] `list.map(f)` applies `f` for `Int` and `String` elements via
      lambda-body inlining (`list_map_strings.can`). Remaining:
      string-element `get`/`first`, nested-map element slot.
- [x] `Bool.and` / `.or` / `.not` chains + bool print newline
      (`bool_chains.can`).
- [x] `?` short-circuits on `Err` in Result-returning functions
      (`try_short_circuit.can`).
- [x] `Int(1)` / `Float(2.5)` / `String("x")` explicit constructors
      (`int_float_explicit_constructors.can`).
- [x] `ToJson` on newtype receivers — method lookup walks the alias
      chain (`tojson_newtype_receiver.can`).
- [x] Option/Result string payloads (was already fixed; pinned by
      `option_string_payload.can`).

**Exit criterion (met 2026-07-02):** the CLAUDE.md "known codegen
gaps" table has no rows where the checker accepts a program that runs
wrong, except the tracked `List<String>.get`/nested-`.map` edge (needs
element-type tracking on `Ty::List`; documented in the table).

### M3 — WASI surface expansion (the Canon ↔ WASI connection)

Depends on M1's encoder routing. Re-generate bindings with the skip
filters removed one class at a time, adding stdlib wrappers + tests:

- [x] Narrow-int signatures, top-level (WIT-informed extern lowering;
      `wasi:cli/exit#exit-with-code` works end-to-end and the stdlib
      `exit` rides it — `tests/exit_code_test.rs`). Narrow ints inside
      records/options (e.g. `system_clock#now`) need compound-shape
      lowering, tracked below.
- [x] `list<string>` returns (`wasi:cli/environment#get-arguments` —
      `Args()` in `canon/std/cli`). Non-string element types
      (`get-random-bytes`'s `list<u8>`) need per-width element
      read-back, the same refactor as `List<String>.get`.
- [x] `option<string>` returns (`get-initial-cwd` — `Cwd()` in
      `canon/std/cli`, pinned by `tests/runtime/cli_args_cwd.can`).
      Bare `result` and non-string options remain.
- [ ] Resources (`own`/`borrow`) — descriptor-based filesystem read /
      write / list-dir as the proving ground.
- [~] Stdlib wrappers: `canon/std/cli` args/cwd/exit done (env-vars
      need `list<tuple<string,string>>`); `canon/std/time` monotonic +
      `canon/std/random` int already ride real WASI bindings;
      `canon/std/fs` needs resources; wall-clock `now` needs record
      returns.
- [ ] `canon install` against a third-party `.wasm` component as an
      integration test — the "bound to a whole ecosystem" claim.

**Exit criterion:** bindgen emits (not skips) the cli, clocks, random,
filesystem, and http surfaces; each wrapper has a runtime or canon-test
fixture.

### M4 — Test plan (runs throughout)

- [x] `TESTPLAN.md`: feature × layer coverage matrix with the holes
      ranked explicitly (H1–H6).
- [x] Thread test failures into the exit code of `canon test` — the
      synthesised main counts failures and drives
      `wasi:cli/exit#exit-with-code`
      (`tests/exit_code_test.rs::canon_test_exit_codes`).
- [ ] Host-integration layer: run built components under real
      `wasmtime` (not just `canon run`) in CI.
- [ ] Negative-path coverage for every M2/M3 feature (checker `fail/`
      fixtures with goldens).
- [ ] `just examples` stays green and becomes part of CI.

**Exit criterion:** coverage matrix has no unexplained holes; CI fails
on any regression in any layer including examples.

### M5 — Flagship example app

An HTTP JSON API service — the domain all recent work points at.

- [x] `examples/notes-api/` with its own `canon.toml` and README:
      path-routed JSON API (index, per-note routes, 404 fallback with
      correct status codes) built from `Request.path()` dispatch and
      `Response(Body * Headers * Status)`.
- [x] Runs under `canon run` (defaults to serving on 127.0.0.1:8080
      when the program has an HTTP entry) and compiles to a standard
      `wasi:http/service` component.
- [ ] Grow it as capabilities land: request method/body introspection,
      JSON request parsing via `?` propagation, response headers, an
      SSE endpoint (STREAMING.md slices 3–4).
- [ ] Referenced from the mdBook as the capstone chapter (M6).

**Exit criterion (core met 2026-07-02):** the README's curl
transcript works on a fresh clone. Remaining scope tracks the open
introspection/composition gaps.

### M6 — Documentation refresh

- [ ] mdBook pass: every tour page verified against current behaviour;
      stdlib reference regenerated to cover the full M3 surface.
- [ ] New guide: "Using WASI interfaces from Canon" (bindgen, manifest
      `[imports]`, `canon install`).
- [ ] New guide: "Deploying Canon components" (`wasmtime serve`, etc.).
- [ ] `DESIGN.md` audit against implementation.
- [ ] Delete `DYNAMIC-HANDLERS.md`; fold closed gaps out of the
      CLAUDE.md table; fix stale claims (e.g. "no external Rust
      dependencies").
- [ ] Capstone chapter walking through the M5 app.

**Exit criterion:** a newcomer can go from install → hello world →
flagship app using only the book.

## Out of scope for V1

- Sockets stdlib wrappers (bindings may exist; no wrapper surface).
- HTTP client beyond what `fetch-url` already demonstrates.
- Package registry / remote `[imports]` sources.
- LSP feature work beyond keeping it compiling.
- Multi-handler HTTP dispatch and inline-lambda handlers.
